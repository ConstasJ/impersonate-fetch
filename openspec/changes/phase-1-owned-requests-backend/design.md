## Context

`impersonated-fetch` currently exposes a Fetch-style Node.js API while delegating TLS/HTTP fingerprinting to prebuilt native artifacts. IDA pseudocode and public repository research show the closed native component is a Go `c-shared` wrapper over `wangluozhe/requests`, which itself uses `wangluozhe/chttp` and uTLS-style ClientHello controls.

Implementation reference material exists under `refs/psuedocodes/`. These files are not production source, but they are accepted as Phase 1 behavioral references for reconstructing the wrapper/shim layer because the project owner extracted them and wants the replacement backend to preserve the observed ABI lifecycle.

Phase 1 is a migration bridge. It prioritizes source ownership, reproducible builds, platform package generation, and ABI parity over architectural slimness. The long-term plan still moves toward a `chttp`-based slim transport in Phase 2 and an authorized Go `net/http` patch stack in Phase 3, but Phase 1 intentionally preserves the current request/stream/free-memory protocol to reduce user-facing risk.

Introducing Go source and release-generated platform backend packages makes Phase 1 a monorepo migration as well as a native backend migration. The repository will move to an Nx-managed pnpm workspace so JavaScript package tasks, Go backend tasks, native artifact generation, and package validation are represented in one task graph. All source packages will reside under `packages/`:
- `packages/impersonated-fetch` - the pure JavaScript/TypeScript main package
- `packages/native-backend` - the unified Go backend source

Cross-platform native builds will use GitHub Actions Matrix to build for Linux (x64, ARM64), macOS (x64, ARM64), and Windows (x64) in parallel.

The current Node native boundary already expects these operations:

- `request(payload)` for buffered responses.
- `streamRequest(payload)` for opening streamed responses.
- `streamRead(streamId, size)` for base64-encoded chunks.
- `streamClose(streamId)` for stream cleanup.
- `free(responseId)` for native response memory cleanup.

## Goals / Non-Goals

**Goals:**

- Build native backend artifacts from source controlled by this repository.
- Preserve current C ABI symbol names and JSON payload compatibility for Phase 1.
- Reproduce current closed-binary behavior closely enough for the existing Node package to switch backend source without public Fetch API changes.
- Add automated contract and differential tests for request, stream, cleanup, and packaging behavior.
- Prepare release-only scoped native npm packages for platform-specific artifacts.
- Convert the repository to an Nx-managed pnpm monorepo with explicit JS package, Go backend, and native package generation project boundaries.
- Keep a single Go backend lineage; Phase 2 will refactor this backend rather than introduce a parallel backend.

**Non-Goals:**

- Do not implement the Phase 2 slim `chttp` transport in this change.
- Do not redesign the public Fetch API.
- Do not move high-level Fetch semantics out of Node in this phase.
- Do not include the separate `chttp` fork repository in this monorepo.
- Do not publish Phase 1 artifacts until the `wangluozhe/requests` license/shipping posture is explicitly approved.

## Decisions

### Decision: Keep the existing ABI for Phase 1

Phase 1 SHALL export the same C ABI symbols currently consumed by the Node native layer. This avoids changing the TypeScript `NativeBindings` contract while build ownership and parity tests are established.

Alternatives considered:

- A new slim ABI now: rejected because it combines backend replacement with architecture redesign.
- Direct `chttp` implementation now: rejected for Phase 1 because it delays removal of the closed binary.

### Decision: Use `wangluozhe/requests` as a transitional implementation bridge

The Phase 1 backend SHALL use `wangluozhe/requests` internally because reverse-engineering evidence shows the closed component already uses this layer. This maximizes parity probability and reduces initial implementation risk.

Alternatives considered:

- Direct `chttp`: better long-term boundary but larger parity gap for initial replacement.
- Direct uTLS: too low-level for Phase 1 because HTTP/2 settings, header order, proxy, cookies, and streaming would need to be rebuilt immediately.

### Decision: Model platform packages as generated release artifacts

The source repository SHALL contain one pure JS/TS main package and one Go backend source package. Platform-specific packages such as `@impersonated-fetch/backend-linux-x64` SHALL be generated by CI/release tooling from templates and compiled artifacts, not hand-maintained as independent source packages.

Alternatives considered:

- Commit every platform package as a normal workspace package: rejected because it creates duplicate metadata and manual maintenance.
- Bundle all binaries in the main package: rejected because it increases install size and weakens platform-specific package validation.

### Decision: Use Nx for Phase 1 monorepo orchestration

Phase 1 SHALL introduce Nx as the task graph and affected-build orchestrator for the mixed Node.js + Go repository. All source packages SHALL be located under `packages/`:
- `packages/impersonated-fetch` - the main JavaScript/TypeScript package
- `packages/native-backend` - the unified Go backend source

Go remains built with ordinary Go commands; Nx wraps those commands, declares inputs/outputs, and coordinates CI/package validation.

#### Decision: Use GitHub Actions Matrix for cross-platform builds

The CI SHALL use GitHub Actions Matrix strategy to build native artifacts in parallel for all supported platforms: Linux (x64, x32, ARM64), macOS (x64, ARM64), and Windows (x64, x32, ARM64). Matrix builds ensure efficient use of CI time and clear separation of platform-specific build concerns.

Alternatives considered:

- Turborepo: rejected for this phase because it is lighter-weight but less suited to explicit graph modeling across generated native artifacts, Go backend tasks, package checks, and future release orchestration.
- pnpm scripts only: rejected because native artifact generation and platform package validation need clearer dependency modeling than flat scripts provide.
- Sequential platform builds: rejected because parallel matrix builds significantly reduce CI time for multi-platform native artifacts.

### Decision: Use the closed binary as an oracle during Phase 1

Tests SHALL compare the self-built backend against the existing closed backend for ABI shape, JSON payload fields, error schema, stream lifecycle, and memory cleanup behavior. Fingerprint smoke tests MAY be network-dependent and separated from deterministic CI tests.

Alternatives considered:

- Trust source similarity only: rejected because parity depends on runtime behavior and native lifecycle details.
- Replace the closed binary immediately without differential tests: rejected because failures would be hard to triage.

### Decision: Use extracted pseudocode as a wrapper behavior reference

Phase 1 implementation SHALL use `refs/psuedocodes/` as a behavioral reference for the C ABI wrapper layer. The implementation should map each exported function to the corresponding pseudocode file:

- `request` → `refs/psuedocodes/main.request.c`
- `stream_request` → `refs/psuedocodes/main.streamRequest.c`
- `stream_read` → `refs/psuedocodes/main.streamRead.c`
- `stream_close` → `refs/psuedocodes/main.streamClose.c`
- `freeMemory` → `refs/psuedocodes/main.freeMemory.c`
- request construction/session behavior → `refs/psuedocodes/main.buildRequest.c` and `refs/psuedocodes/main.GetSession.c`

The pseudocode should guide observable lifecycle details: JSON field names, error string prefixes, response id generation and storage, stream pool behavior, read chunk encoding, EOF semantics, and pointer cleanup keys. It should not override stronger evidence from runnable differential tests.

Alternatives considered:

- Ignore the pseudocode and infer behavior only from public `requests` APIs: rejected because the exported C ABI lifecycle includes wrapper-specific memory and stream-pool behavior.
- Treat pseudocode as exact source to mechanically port: rejected because the implementation should be idiomatic Go and verified by behavior tests rather than a line-by-line decompilation clone.

### Decision: Keep `chttp` as an external dependency path

The `chttp` fork and future authorized patch stack SHALL be maintained outside this monorepo. Phase 1 may depend on transitive `chttp` through `wangluozhe/requests`, but direct `chttp` repository ownership is out of scope for this change.

## Risks / Trade-offs

- Phase 1 keeps the duplicate Go HTTP-client layer temporarily → Mitigation: mark the requests-based backend as transitional and keep Phase 2 in the roadmap.
- `wangluozhe/requests` has GPL-3.0 licensing implications → Mitigation: require an explicit license/shipping decision before public release and keep Phase 1 artifacts unreleased if the decision is unresolved.
- Differential tests may expose behavior differences in cookies/session handling or stream cleanup → Mitigation: capture those differences as compatibility decisions before switching default backend source.
- Pseudocode may contain decompiler artifacts or misleading control flow → Mitigation: use it as a behavioral checklist and confirm with oracle tests before treating details as required compatibility behavior.
- Cross-compilation can produce platform-specific dynamic library differences → Mitigation: add per-target package checks and keep native artifact filenames/platform mapping explicit.
- Generated native npm packages can drift from the main package metadata → Mitigation: generate package metadata from a single template and verify package contents in CI.
- Nx adoption can add configuration overhead → Mitigation: keep Nx usage limited to project boundaries, task orchestration, affected checks, cache outputs, and release/package-generation tasks; do not replace Go tooling internals.

## Migration Plan

1. Add Go backend source at `packages/native-backend` that exports the existing C ABI and uses `wangluozhe/requests` internally.
2. Convert the repository into an Nx-managed pnpm monorepo with projects at `packages/impersonated-fetch` (JS package), `packages/native-backend` (Go backend), and native package generation workflow.
3. Add local build tasks for the host platform, then expand to the supported platform matrix.
4. Set up GitHub Actions Matrix for parallel cross-platform builds (Linux x64/x32/ARM64, macOS x64/ARM64, Windows x64/x32/ARM64).
5. Add ABI contract tests against the self-built backend.
6. Add differential oracle tests comparing the self-built backend with the existing closed backend.
7. Add generated scoped native package artifacts and package validation.
8. Switch local development and CI to prefer the self-built backend only after parity gates pass.
9. Keep the closed backend available as an oracle/fallback until Phase 1 release criteria are met.

Rollback strategy: keep current closed native artifacts and existing loader behavior available until the self-built backend passes parity, packaging, and license gates.
