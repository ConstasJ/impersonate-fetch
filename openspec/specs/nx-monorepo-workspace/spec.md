## Requirements
### Requirement: Nx-managed monorepo workspace
The system SHALL use Nx with pnpm workspaces to coordinate the pure JavaScript package, unified Go backend source, and release-time native package generation tasks. All source packages SHALL reside under `packages/` directory.

#### Scenario: Nx workspace is configured
- **WHEN** contributors inspect the repository root after Phase 1 setup
- **THEN** they find Nx configuration that defines project boundaries for:
  - `packages/impersonated-fetch` - the pure JavaScript/TypeScript main package
  - `packages/native-backend` - the unified Go backend source
  - native package generation workflow

#### Scenario: pnpm workspace includes source packages
- **WHEN** pnpm workspace discovery runs
- **THEN** it includes:
  - `packages/impersonated-fetch` - the source-maintained JavaScript package
  - `packages/native-backend` - the Go backend source package
  - tooling packages under `packages/*`
- **AND** generated platform backend packages are NOT required to be committed as normal source workspaces

### Requirement: Pure JavaScript main package boundary
The monorepo SHALL keep the main `impersonated-fetch` npm package as a pure JavaScript/TypeScript package that consumes generated backend packages at install/runtime.

#### Scenario: Main package project builds
- **WHEN** the Nx build target for the main package runs
- **THEN** it builds the TypeScript package without compiling Go code as part of the package build target

#### Scenario: Main package depends on native package artifacts
- **WHEN** release packaging runs
- **THEN** the main package can declare optional platform backend package dependencies generated by the release workflow

### Requirement: Unified Go backend project boundary
The monorepo SHALL define exactly one source-maintained Go backend project located at `packages/native-backend` for Phase 1, with Phase 2 implemented as a refactor of the same backend lineage rather than a parallel backend package.

#### Scenario: Go backend project is defined
- **WHEN** contributors inspect the Nx project graph
- **THEN** they see one Go backend source project at `packages/native-backend` responsible for building the native backend artifacts

#### Scenario: Phase 2 is represented as refactor path
- **WHEN** contributors inspect Phase 1 documentation or project metadata
- **THEN** they can see that Phase 2 removes `wangluozhe/requests` from the same backend lineage rather than introducing a second production backend

### Requirement: GitHub Actions Matrix for cross-platform builds
The system SHALL use GitHub Actions Matrix strategy to build the Go native backend in parallel across all supported platforms: Linux (x64, x32, ARM64), macOS (x64, ARM64), and Windows (x64, x32, ARM64).

#### Scenario: CI builds all platform targets
- **WHEN** the CI workflow runs for the Go backend
- **THEN** it uses a matrix strategy to spawn parallel jobs for each supported platform/architecture combination
- **AND** each job produces platform-specific artifacts

#### Scenario: CI artifacts are uploaded for release
- **WHEN** cross-platform builds complete
- **THEN** all platform artifacts are uploaded to workflow artifacts or release assets
- **AND** the release workflow can consume these artifacts to generate scoped backend npm packages

### Requirement: Release-time generated backend packages
The monorepo SHALL treat scoped platform backend npm packages as generated release artifacts, not hand-maintained source packages.

#### Scenario: Native package generation target runs
- **WHEN** the Nx native package generation target runs after native builds complete
- **THEN** it creates scoped platform package directories from templates and compiled artifacts

#### Scenario: Generated packages are not source packages
- **WHEN** contributors inspect normal source workspace packages
- **THEN** generated platform backend packages are absent unless a release/package-generation task has produced them

### Requirement: Graph-aware task orchestration
The monorepo SHALL use Nx task orchestration for build, test, lint, typecheck, Go build/test, native package generation, and package validation targets.

#### Scenario: Affected tasks run
- **WHEN** CI runs affected checks for a change touching only JavaScript package files
- **THEN** Nx runs the affected JavaScript targets without unnecessarily rebuilding unrelated native artifacts

#### Scenario: Native changes trigger dependent checks
- **WHEN** CI runs affected checks for a change touching Go backend source or native package generation tools
- **THEN** Nx runs the relevant Go backend build/test targets and dependent package validation targets

#### Scenario: Cache outputs are declared
- **WHEN** Nx runs cacheable build targets
- **THEN** JavaScript build output, Go build output, and generated native package artifacts use declared output paths suitable for local and CI caching
